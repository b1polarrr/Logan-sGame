Redis 学习笔记：从入门到实战

Redis（Remote Dictionary Server）是一款开源的内存键值数据库，以极高的读写性能著称。 本文系统梳理 Redis 的核心概念、数据类型、持久化策略与缓存实践，适合后端开发者入门与复习。

阅读约 15 分钟 难度：入门 → 进阶

1. Redis 是什么

Redis 由 Salvatore Sanfilippo 于 2009 年发布，基于 C 语言编写， 数据全部存储在内存中，因此单线程模型下也能达到 10 万+ QPS 的读写吞吐。 它支持多种数据结构，并提供持久化、主从复制、哨兵、集群等企业级特性。

典型应用场景

  • 缓存：减轻 MySQL 等关系型数据库的读压力，是最常见的用法
  • 会话存储：保存用户登录态（Session / Token）
  • 排行榜：利用 Sorted Set 实现实时排名
  • 消息队列：List / Stream 实现简单的生产者-消费者模型
  • 分布式锁：SET key value NX EX ttl 实现互斥
  • 限流计数：滑动窗口、固定窗口计数器
💡 为什么选 Redis 而不是 Memcached？
Memcached 只支持简单的 KV 字符串，且没有持久化。 Redis 提供丰富的数据结构、持久化、发布订阅、Lua 脚本等能力，功能更全面，生态也更活跃。

单线程模型

Redis 6.0 之前，网络 I/O 与命令执行均在单线程中完成。 这意味着不需要加锁，避免了上下文切换开销，保证了命令的原子性。 Redis 6.0 引入了多线程 I/O（仅负责网络读写），命令执行仍是单线程。

┌─────────────┐ 网络请求 ┌──────────────────────┐ │ Client │ ──────────────▶ │ Redis Server │ │ (Java/Go) │ ◀────────────── │ ┌────────────────┐ │ └─────────────┘ 响应 │ │ 事件循环 │ │ │ │ (epoll/kqueue)│ │ │ └───────┬────────┘ │ │ ▼ │ │ ┌────────────────┐ │ │ │ 命令处理器 │ │ │ │ (单线程) │ │ │ └───────┬────────┘ │ │ ▼ │ │ ┌────────────────┐ │ │ │ 内存数据结构 │ │ │ └────────────────┘ │ └──────────────────────┘

2. 安装与启动

Docker 快速启动（推荐）

BASH
docker run -d \
  --name redis \
  -p 6379:6379 \
  redis:7-alpine \
  redis-server --appendonly yes

连接测试

REDIS CLI
# 进入容器
docker exec -it redis redis-cli

# 测试连通性
127.0.0.1:6379> PING
PONG

# 设置并读取
127.0.0.1:6379> SET name "Redis Learner"
OK
127.0.0.1:6379> GET name
"Redis Learner"

Java 客户端（Lettuce）

JAVA
// Maven 依赖
// <dependency>
//   <groupId>io.lettuce</groupId>
//   <artifactId>lettuce-core</artifactId>
//   <version>6.3.2.RELEASE</version>
// </dependency>

RedisClient redisClient = RedisClient.create("redis://localhost:6379");
StatefulRedisConnection<String, String> connection = redisClient.connect();
RedisCommands<String, String> commands = connection.sync();

commands.set("player:1001", "Alice");
String playerName = commands.get("player:1001"); // "Alice"

connection.close();
redisClient.shutdown();

3. 五种基本数据类型

Redis 的核心优势在于丰富的内置数据结构，每种类型对应不同的使用场景。

类型 说明 典型场景
String 二进制安全的字符串，最大 512 MB 缓存对象 JSON、计数器、分布式锁
Hash 字段-值映射表 存储用户信息、商品属性
List 双向链表，有序可重复 消息队列、最新动态列表
Set 无序集合，元素唯一 标签、共同好友、去重
Sorted Set 有序集合，每个元素带 score 排行榜、延迟队列

String — 字符串

REDIS CLI
SET counter 0
INCR counter          # 原子自增 → 1
INCRBY counter 10     # 加 10 → 11
DECR counter          # 自减 → 10

SET session:abc123 "userId=42" EX 3600   # 3600 秒后过期
SET lock:order NX EX 30                    # 分布式锁，30 秒超时

Hash — 哈希表

REDIS CLI
HSET user:1001 name "Alice" age 25 city "Shanghai"
HGET user:1001 name          # "Alice"
HGETALL user:1001            # 返回所有字段
HINCRBY user:1001 age 1      # age → 26

List — 列表

REDIS CLI
LPUSH queue:task "task-A" "task-B"   # 左侧入队
RPOP queue:task                       # 右侧出队 → "task-A"
LRANGE queue:task 0 -1               # 查看全部元素

Set — 集合

REDIS CLI
SADD tags:article:1 "Java" "Redis" "Backend"
SISMEMBER tags:article:1 "Redis"   # 1（存在）
SINTER tags:article:1 tags:article:2  # 求交集

Sorted Set — 有序集合

REDIS CLI
ZADD leaderboard 9800 "player_A" 8500 "player_B" 9200 "player_C"
ZREVRANGE leaderboard 0 2 WITHSCORES   # 前三名
# 1) "player_A"  2) "9800"
# 3) "player_C"  4) "9200"
# 5) "player_B"  6) "8500"
ZINCRBY leaderboard 200 "player_B"     # 加分

4. 常用命令速查

键（Key）管理

REDIS CLI
KEYS user:*          # 查找匹配的键（生产环境慎用，阻塞）
SCAN 0 MATCH user:* COUNT 100   # 渐进式扫描，推荐
EXISTS user:1001     # 判断键是否存在
DEL user:1001        # 删除键
EXPIRE user:1001 60  # 设置 60 秒过期
TTL user:1001        # 查看剩余生存时间（秒）

事务

Redis 事务通过 MULTI / EXEC 实现命令的批量执行，但不支持回滚。

REDIS CLI
MULTI
SET account:A 900
SET account:B 1100
EXEC
⚠️ 注意
如果事务中某条命令语法错误，整个事务不会执行； 若运行时错误（如对 String 执行 LPUSH），其余命令仍会执行。 需要条件判断时使用 Lua 脚本或 WATCH 乐观锁。

发布 / 订阅

REDIS CLI
# 终端 A（订阅者）
SUBSCRIBE room:666

# 终端 B（发布者）
PUBLISH room:666 "玩家 Alice 加入了房间"

5. 持久化机制

内存数据在重启后会丢失，Redis 提供两种持久化方式将数据写入磁盘。

RDB（快照）

在指定时间间隔内，将内存数据的快照写入 dump.rdb 文件。 恢复速度快，但两次快照之间的数据可能丢失。

REDIS.CONF
save 900 1      # 900 秒内至少 1 次写操作
save 300 10     # 300 秒内至少 10 次
save 60 10000   # 60 秒内至少 10000 次

AOF（追加文件）

将每条写命令追加到 appendonly.aof 文件。 数据完整性更高，但文件体积较大，恢复速度略慢。

REDIS.CONF
appendonly yes
appendfsync everysec   # 每秒 fsync 一次（推荐）
# appendfsync always   # 每次写都 fsync，最安全但最慢
# appendfsync no       # 由 OS 决定，最快但可能丢数据
💡 生产建议
通常同时开启 RDB + AOF：AOF 保证数据完整性，RDB 用于快速冷启动备份。 定期将 RDB 文件复制到异地存储。

6. 缓存设计模式

在后端系统中，Redis 最常见的角色是缓存层，位于应用与数据库之间。

请求 缓存命中 直接返回 Client ──▶ App ──▶ Redis ──────────────────────▶ Client │ ▲ │ 未命中 │ 写入缓存 ▼ │ MySQL ──────┘

Cache-Aside（旁路缓存）

最常用的模式，应用程序自行管理缓存与数据库的读写。

  • 读：先查 Redis，miss 则查 DB，再写入 Redis
  • 写：先更新 DB，再删除（而非更新）Redis 中的缓存
JAVA 伪代码
public User getUserById(long userId) {
    String cacheKey = "user:" + userId;
    String cached = redis.get(cacheKey);
    if (cached != null) {
        return JSON.parse(cached, User.class);
    }
    User user = userRepository.findById(userId);
    if (user != null) {
        redis.setex(cacheKey, 3600, JSON.stringify(user));
    }
    return user;
}

public void updateUser(User user) {
    userRepository.save(user);
    redis.del("user:" + user.getId());  // 删除缓存，非更新
}

三大经典问题

问题 描述 解决方案
缓存穿透 查询不存在的数据，每次都打到 DB 缓存空值；布隆过滤器
缓存击穿 热点 Key 过期瞬间，大量请求涌向 DB 互斥锁；逻辑过期（不设置 TTL）
缓存雪崩 大量 Key 同时过期，DB 压力骤增 过期时间加随机偏移；多级缓存；限流降级

7. 高可用与集群

主从复制（Replication）

一个 Master 负责写，多个 Slave 负责读，Master 将写命令异步同步给 Slave。 Master 宕机后需手动或通过哨兵提升某个 Slave 为新 Master。

REDIS CLI
# 在 Slave 节点执行
REPLICAOF 192.168.1.10 6379

哨兵（Sentinel）

哨兵是一组独立进程，监控 Master 和 Slave 的健康状态。 当 Master 不可达时，哨兵会自动进行故障转移（Failover）， 选举新的 Master 并通知客户端。

Cluster 集群

Redis Cluster 将数据划分为 16384 个哈希槽（slot）， 分布在多个 Master 节点上，每个 Master 可挂载 Slave。 客户端通过 CRC16(key) mod 16384 计算 slot，路由到对应节点。

BASH
# 创建 3 主 3 从集群
redis-cli --cluster create \
  127.0.0.1:7001 127.0.0.1:7002 127.0.0.1:7003 \
  127.0.0.1:7004 127.0.0.1:7005 127.0.0.1:7006 \
  --cluster-replicas 1

8. 实战练习

以下练习由浅入深，建议在本地 Docker Redis 环境中动手完成。

练习 1：实现一个简单的计数器

用 String 类型实现文章阅读量统计，每次阅读 INCR page:views:123。

练习 2：排行榜

用 Sorted Set 维护游戏积分榜，实现 Top 10 查询与加分操作。

练习 3：分布式锁

REDIS CLI
# 加锁（NX = Not eXists，EX = 过期秒数）
SET lock:order:1001 "thread-42" NX EX 30

# 释放锁（需验证 value 是自己的，防止误删）
# 推荐用 Lua 脚本保证原子性
EVAL "if redis.call('get', KEYS[1]) == ARGV[1] then
        return redis.call('del', KEYS[1])
      else return 0 end" 1 lock:order:1001 thread-42

练习 4：Session 缓存

模拟 Web 应用登录：登录成功后 SET session:{token} {userId} EX 7200， 每次请求携带 token，从 Redis 读取 userId，实现无状态服务的会话管理。
